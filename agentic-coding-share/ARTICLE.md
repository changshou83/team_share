## 工作流

官方推荐的工作流：Explore -> Plan -> Code -> Commit。

1. Explore + Plan：开始编码前，使用Plan Mode与模型对话，确定一个修改方案。
2. Code + Commit：让Agent按照确定好的方案进行编码，开发人员验收后，将其提交到Git仓库，并开启下一个工作流。

我的工作流：

1. UI生成：先使用Pixso MCP读取设计稿链接，再加上页面的功能描述，生成UI初稿。然后再根据需要逐步细化，如果三次调整达不到预期就改为手写实现。
2. 功能实现：对于比较大块的功能会新建一个对话（保持干净上下文），并开启Plan Mode，将需求转换为多条确定的规则，可以提高一次完成率。

## AI Coding 工具使用体验

Cursor: 
（1）支持多模型，可以做到让好的模型规划，差一些的模型执行
（2）可以是否应用模型的修改
（3）不需要再多开一个IDE，可以在一个工具里完成开发工作
（1）但是感觉没有Claude Code和Codex智能。
Claude Code: 
（1）我使用的是DeepSeek V4，执行简单的任务时，效果还算可以
（2）执行流程很清晰，能知道他当前在做什么，方便及时介入
（3）直接开一个命令行就能用，很方便。
Codex App: 
（1）我使用的是GPT 5.4/5.5 high，执行中等难度的任务时，效果还算可以
（2）并行执行任务体验较好，不像Claude Code还需要再开一个命令行
（3）但是上下文窗口比较小（258K），运行长任务的时候容易触发上下文自动压缩。

最大的感受：模型能力决定下限，上下文质量决定上限。

## 注意事项

（一）UI生成
生成初稿时，需要说明是图片或图标的元素，可以先生成占位元素，后续再手动切图并进行替换。
生成样式时，需要明确指明生成嵌套关系的样式，并且在生成后需要检查UI的布局是否符合预期，例如元素大小，颜色值，布局等。
（二）代码质量要求
只改动必要的部分，优先复用现有成熟代码，避免重复造轮子。
架构设计时让边界情况自然融入常规逻辑，而不是单独打补丁。
保持代码简单直观，不过度设计复杂架构方案。
代码应表达实际逻辑，结构清晰，不保留不再使用的代码，不留无用的混淆项，避免未来维护困惑。
在代码文件中使用与用户沟通一致的语言撰写简明专业的代码注释。
（三）其他
保持专注：建议同时处理的对话数量不超过两个，并尽量保持高相关性，目的为保持专注，降低频繁上下文切换带来的精力消耗。 
保持干净上下文：在对话中尽量保持上下文含义相同，不要所有的问题都堆在一个对话中。
